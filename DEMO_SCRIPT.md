## Employer Q&A — Prepared Answers

---

**Q: Why XGBoost instead of a neural network or collaborative filtering?**

> "Three reasons. Speed — XGBoost inference is microseconds, neural nets need
> GPU warmup. Explainability — TreeSHAP is mathematically exact for tree models;
> for neural nets you'd need approximate methods like LIME which are slower
> and less reliable. And memory — XGBoost runs in 150MB; a PyTorch model would
> be ten times that on a 2GB server."

---

**Q: What is SHAP exactly?**

> "SHAP stands for SHapley Additive exPlanations. It's from cooperative game theory —
> it assigns each feature a contribution value that fairly represents how much
> that feature was responsible for the prediction, relative to a baseline.
> For tree models, TreeExplainer computes exact SHAP values in polynomial time —
> no approximation needed. The values sum to the model output minus the expected
> output, which is why they're mathematically meaningful."

---

**Q: How do you handle it when GPT-4o-mini is down?**

> "The LLM explainer wraps every OpenAI call in a try/except that catches
> any exception — network error, rate limit, invalid key, empty response.
> It then generates a fallback explanation from the SHAP reasons using a
> template. The user sees a coherent two-sentence explanation either way.
> The system has never returned an error to the frontend for an explanation request."

---

**Q: Is this production-ready or just a demo?**

> "It's running in production right now. The backend has a health endpoint,
> request logging middleware, a global exception handler that never leaks
> stack traces, async database sessions with connection pooling, four TTL
> caches, and it's containerised with Docker. The frontend is on Vercel's CDN.
> The only thing that's not production-scale is the dataset size — 50,000 reviews
> is enough to demonstrate the system, but the architecture handles millions."

---

**Q: What would you add with more time?**

> "Real collaborative filtering with a user-item matrix, A/B testing different
> ranking strategies with the dashboard tracking which performs better,
> and actual pgvector semantic search using OpenAI embeddings — the column and
> index are already in the schema, I just didn't want to spend $50 embedding
> 50,000 products before the showcase."

---

**Q: Walk me through the tech stack.**

> "Backend: FastAPI on Python 3.11, XGBoost for ranking, SHAP TreeExplainer
> for explanations, LangChain + GPT-4o-mini for natural language summaries.
> Database: Supabase PostgreSQL with pgvector, accessed via SQLAlchemy async
> with asyncpg. Caching: cachetools TTLCache, four separate caches with
> different TTLs. Infrastructure: DigitalOcean Droplet in Frankfurt, 2GB RAM,
> always-on — covered by GitHub Student Pack credit. Frontend: React 18 + Vite,
> TailwindCSS, Framer Motion animations, Recharts for all visualisations,
> Zustand for state, React Query for data fetching, deployed on Vercel."

---

**Q: How did you evaluate the model?**

> "I used NDCG@10 — Normalised Discounted Cumulative Gain at rank 10.
> It's the standard metric for learning-to-rank problems because it penalises
> putting relevant items lower in the list more than putting them higher.
> I also computed Precision@5 — what fraction of the top 5 recommendations
> are actually relevant. I did a group-aware train/test split so users in the
> test set were never seen during training."

---

**Q: Why not use a pre-built recommendation library?**

> "I could have used Surprise or LightFM, but building from scratch meant I
> could instrument every step — custom features, custom SHAP integration,
> custom caching strategy. If I'd used a black-box library the explainability
> layer would have been much harder to implement correctly."

---

**Q: How does the personalisation actually work?**

> "For each request I query the user's interaction history from the database —
> their average rating, review count, favourite category, and recency of activity.
> I then build a feature vector for every candidate product combining those
> user stats with product stats. The XGBoost ranker scores all 50 candidates
> and returns the top 5. The same model serves all users — the personalisation
> comes from the features, not separate models per user."

---

**Q: What's the cost to run this?**

> "The Droplet is $12 a month, covered by the $200 GitHub Student Pack credit.
> Supabase is free tier. Vercel is free tier. OpenAI is hard-capped at $10 a month —
> realistically it's under $1 because explanations are cached for an hour,
> so the same product is only explained once per hour across all users."
